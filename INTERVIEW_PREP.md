# 面试备战文档（医疗智能客服 RAG 系统）

## 一、项目一句话介绍
基于 LangGraph + vLLM 的医疗健康智能问答系统，采用“规则优先 + RAG 增强 + 上下文工程”，解决医疗数值歧义与检索幻觉问题，并支持 SSE 流式输出与会话记忆。

## 二、技术栈清单（按模块）
- 服务框架：FastAPI + Uvicorn（SSE 支持、REST API）
- LLM 推理：vLLM（OpenAI-compatible API）
- 任务编排：LangGraph（状态机式 Agent 流程）
- 检索：FAISS 向量检索 + TF-IDF 兜底
- 规则层：自研规则引擎（数值/关键字规则）
- 知识图谱：轻量 KG 检索（图谱命中补充召回）
- 记忆：短期会话缓存（Redis 可选）+ 记忆摘要（DB 可选）
- 评测：规则命中测试、检索评测脚本
- 前端：Gradio

## 三、架构与流程（面试口述版）
1. 用户请求进入 FastAPI。
2. LangGraph 编排：load_session → risk → rule_extract → retrieve → rerank → hypothesis → generate → save_session。
3. 规则层先识别数值并分流，结构化字段写入 prompt 的规则区。
4. RAG 检索：FAISS + KG；规则证据优先进入 evidence。
5. 生成层强制引用证据，SSE 先发证据再流式输出答案。

## 四、关键设计亮点
- 规则优先：解决“38℃ vs 40℃”差异与数值歧义。
- 上下文工程：规则区 + 证据区 + 历史摘要 + 当前问题，避免上下文膨胀。
- SSE 流式输出：证据先发，避免证据链错位。
- 检索兜底：embedding 失败时 TF-IDF fallback，避免“证据为空”。

## 五、遇到的工程问题与解决方案
1. 数值歧义导致检索错误
   - 问题：发烧 38℃ 被检索成“血小板38”。
   - 解决：规则层先抽取 temp_c；规则证据置顶；查询归一化（发热 体温 38℃）。

2. 规则命中不生效
   - 问题：规则解析为空，rule_hits 不触发。
   - 解决：排查发现请求体 UTF-16 导致 query 乱码；改为 UTF-8 bytes 发送。

3. Embedding 模型加载失败（WinError 1455）
   - 问题：torch DLL 加载失败，检索无证据。
   - 解决：TF-IDF fallback；必要时升级页面文件或改用 CPU torch。

4. SSE 连接断流
   - 问题：前端流式解析不稳定。
   - 解决：标准 SSE 解析；流式失败自动回退非流式。

5. 语料乱码
   - 问题：原始数据编码导致乱码。
   - 解决：强制 UTF-8 入库；验证显示编码；PowerShell 输出用 UTF-8。

6. meta 与 index 不一致
   - 问题：meta_qa.jsonl 行数与索引向量不匹配。
   - 解决：重建 meta 文件，确保一致性。

## 六、常见面试问题与回答要点
1. RAG 幻觉如何定位？
   - 用两维判断：相关性（检索）+ 忠实度（生成）。
   - 证据不相关但回答忠实 → 检索幻觉；证据相关但回答乱编 → 生成幻觉。

2. 规则层的价值是什么？
   - 用于数值差异分流（38 vs 40）与安全兜底。
   - 结构化字段直接进入 prompt，指导模型而非替代检索。

3. 如何做上下文工程？
   - 规则区/证据区/历史摘要/当前问题分区。
   - 历史过长先摘要，再滑动窗口。

4. SSE 为什么要先发证据？
   - 防止证据链错位；保证生成过程只引用已锁定 evidence。

5. TF-IDF 兜底是不是降低精度？
   - 是兜底保证“有证据”；线上可监控并在资源充足时恢复 embedding 检索。

## 七、你能强调的量化点（可选）
- 规则命中率（rule hit rate）
- Recall@K（检索命中率）
- 证据一致性（Cited Evidence %）

## 八、复盘与成长点
- 规则层 + RAG 的协作比单纯检索更稳。
- 工程调试重点：编码、索引一致性、依赖版本、流式稳定性。
